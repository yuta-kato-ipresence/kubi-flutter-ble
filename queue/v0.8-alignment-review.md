# kubi-flutter-ble v0.8 整合レビュー（チェックリスト正本）

> **目的**: Phase 3 実装着手前に、TS 側 `kubi-web-ble` v0.8 (Unreleased) と Flutter 側の現状（README / docs/api-design.md / lib/src/** / example）の乖離を全て洗い出し、項目ごとに採用/棄却/Flutter アレンジを決める。

各項目の `[ ]` を `[x]` に更新しながら議論を進める。`Decision` 欄を埋めてから設計書・実装に反映する。

### A1. UUID 定数の全面ズレ（`lib/src/kubi_protocol.dart`）

旧 `kubi_flutter_plugin` から誤って継承した疑い。TS 側 `src/constants.ts` の `GATT_UUIDS` が実機検証済み一次ソース。

| 用途 | TS (実機検証済) | Flutter 現状 | 状態 |
|---|---|---|---|
| PAN goal | `00009145-...` | `00009143-...` | **誤** |
| TILT goal | `00009146-...` | `00009144-...` | **誤** |
| MOTOR_POSITION (notify) | `00009147-...` | `00009145-...` | **誤** |
| REG_READ_1BYTE | `00009143-...` | 定義なし | **欠落** |
| REG_READ_2BYTE | `00009144-...` | 定義なし | **欠落** |
| PANTILT_CONFIG | `00009142-...` | `00009142-...` | OK |
| SERVO_SERVICE | `2a001800-...` | `servoServiceUuid` 同値 | OK |

- [x] **A1: 即修正（議論不要）** — `panUuid`/`tiltUuid`/`motorPositionUuid` を正しい値へ、`regRead1ByteUuid`/`regRead2ByteUuid` を追加

### A2. `kubiServiceUuid 0000e001` / `ledUuid 0000e002` の出所不明

TS 側 `GATT_UUIDS` に存在しない。実機検証ソース (`docs/servo-spec.md`) でも `2a001800-...` のみが primary service。

- [x] **A2**: スタブ実装の誤りと確定 → `kubiServiceUuid` / `ledUuid` を削除。  
  根拠: ① `0000e002` はどの一次ソース (kubi-ble `constants.ts` / `servo-spec.md` / 旧 `kubi_flutter_plugin/request.md` §8.6 / 旧 `kubi_flutter_plugin/lib/src/kubi_protocol.dart`) にも存在しない誤記。② 実機の LED は `0000e104` (Active Indicator Color, RGB 3byte)。③ `0000e001` は実在の Custom Status service だが battery/error/button/indicator-color を扱うものでスコープ外。将来 LED/battery を扱う際は servo-spec.md §8.6 準拠で別途定義する旨を `kubi_protocol.dart` にコメント残置。
- Decision: _未決_（`kubi_flutter_plugin` 側の使用箇所を確認してから判断）

### A3. `KubiProtocol.parsePosition(List<int>)` の前提が間違い

現実装: `data[0..1]=panBE`, `data[2..3]=tiltBE` の 1 フレーム解釈。
TS 実機検証済プロトコル: `0x9147` notify は **register read 応答**。`bytes[3]` = lo、`bytes[4]` = hi (LE)。1 notify = 1 軸 1 register。pan+tilt 同時返却フレームは存在しない。

- [ ] **A3**: `parsePosition` は削除し、`parseRegisterReadResponse(bytes, byteWidth)` 相当に置き換える。Phase 3 で `_readRegister` 実装と一体で書き直す。
- Decision: 削除 + 置換（議論不要）

### A4. `servoAngle(double deg)` が内部で `clampPan` を呼ぶ

- [x] **A4: 即修正（議論不要）** — `servoAngle` を軸非依存の `[-150, 150]` クランプに直す。`clampPan` 経由をやめる。  
  → `lib/src/kubi_protocol.dart` 修正済 (TS `web-kubi-ble.ts` と同等)
- Decision: 修正

### A5. 謎の `static const double defaultSpeed = 52.3`（`kubi_protocol.dart:30`）

TS にも仕様書にも該当しない。出所不明。`defaultMoveSpeed = 100` とは別物として残っている。

### A5.b 🆕 速度算出ロジックが未実装（A5 削除に伴い顕在化）

`52.3` を削除した結果、Flutter 側に「ユーザー指定の double `speed` をどう servo register 値 (`servoSpeed` の引数 int 1..100) に丸めるか」のロジックが**完全に欠落**していることが露呈した。

- TS 側の正本: `web-kubi-ble.ts:140` `clampSpeed(speed)` = `Math.max(1, Math.min(100, Math.round(toFinite(speed ?? DEFAULT_MOVE_SPEED))))`  
  - 入力 `number | undefined`, 出力 `1..100` 整数
  - `undefined`/非有限 → 100 fallback
  - `0` や `0.4` → 1 にクランプ（servo 仕様 §8.2: 0 は無限速の特殊値で意図せず踏まない方針）
  - `52.3` → `Math.round` で 52
- Flutter 側現状: `KubiProtocol.servoSpeed(int? speed)` は `int? → int` で `null → 100`、`< 1 → 1`、`> 100 → 100` のみ。**double 入力経路 / 丸め / 非有限処理がない。**
- 設計書 `docs/api-design.md` §3.2 では `MoveSpeed = int | ({int pan, int tilt})` 想定だが、TS は `MoveSpeed = number | ...` で double 受け入れ可。Flutter 側を int に閉じるか double を受け入れるかは API 設計の話 → B 系で議論。
- どちらにせよ「`clampSpeed` 相当の正規化関数」は必須。

- [ ] **A5.b**: `KubiProtocol.clampSpeed` (または `normalizeSpeed`) を追加し、TS と挙動を一致させる（`Math.round` 相当・1..100 clamp・null/非有限 fallback）。
- [ ] B 系で「`MoveSpeed` を `int` に閉じるか `num`/`double` を受け入れるか」を確定後、本 issue 実装。
- Decision: _未決（B 系決着待ち）_

## B. TS v0.8 機能追従漏れ（同等機能要求）

### B1. `connect(device, {Duration timeout})` の default と境界値の扱い

TS: `timeoutMs` 省略/0/負値/非数 → `KUBI_CONSTANTS.CONNECTION_TIMEOUT_MS` (= 3000) フォールバック。`Promise.race` で `BleConnectionError("Connection timeout after Nms")` reject。

- [x] **B1 決定**: TS の defensive fallback (`>0 でなければ default`) は JS の弱い型システム由来なので踏襲しない。**案 1 採用**:
  - シグネチャ: `Future<void> connect(KubiDevice device, {Duration timeout = const Duration(milliseconds: 3000)})`
  - `null` 廃止 (non-null + default value)
  - `timeout.inMicroseconds <= 0` (zero/負値) → `ArgumentError`
  - timeout 超過 → `BleConnectionError("Connection timeout after ${ms}ms")` (TS と同メッセージ・例外型)
- 影響:
  - `docs/api-design.md` §3.1 を更新 (`Duration?` → `Duration`、default 値明記)
  - `lib/src/kubi_ble.dart:35` を更新 (現状 `Future<void> connect(KubiDevice device);` → 上記シグネチャ)
  - `KubiProtocol.connectionTimeoutMs` (定数) は維持 (`Duration` リテラルは `const` だが、内部実装で参照する場合のため)
- Decision: 案 1 (non-null default + ArgumentError、コード反映は B 系一括で)

### B2. `DisconnectReason` 6 値の追加（破壊的）

TS: `'user' | 'timeout' | 'device-lost' | 'error' | 'reconnect-exhausted' | 'unknown'`。`onConnectionStateChange(state, reason?)` の第 2 引数で配信。

- [x] **B2 決定**: 案 3 採用 — `Stream<ConnectionStateEvent>` (immutable class) で配信。
  ```dart
  enum DisconnectReason {
    user,                 // disconnect() による意図的切断
    timeout,              // connect() のタイムアウト
    deviceLost,           // OS からの disconnect 通知 (universal_ble の onConnectionChange false)
    error,                // 接続中に例外発生
    reconnectExhausted,   // 自動再接続リトライ上限到達
    unknown,              // 後方互換フォールバック
  }

  final class ConnectionStateEvent {
    const ConnectionStateEvent({
      required this.state,
      this.reason,
      required this.timestamp,
    });
    final BleConnectionState state;
    final DisconnectReason? reason;  // 切断・失敗時のみ非 null (state == disconnected/disconnecting)
    final DateTime timestamp;
    // == / hashCode / toString は C4 (equatable 採用可否) で確定
  }
  ```
- 影響:
  - `docs/api-design.md` §3.1: `Stream<BleConnectionState>` → `Stream<ConnectionStateEvent>`、§8.1 に `DisconnectReason` / `ConnectionStateEvent` 追加
  - `lib/src/kubi_ble.dart`: コールバック型 `void Function(BleConnectionState)` を廃止し `Stream<ConnectionStateEvent> get connectionStateStream` に移行 (現状 callback ベースから Stream への破壊的変更)
  - 現在状態の即時参照は `BleConnectionState get currentConnectionState` を維持 (Stream 購読時の初期値問題は C6 で別途議論)
- 理由: B7/B9/B13/B14 と同様、v0.8 はイベント/ペイロードを class で統一する方向。Record より == / Dartdoc / 拡張性で勝る。
- Decision: 案 3 (`ConnectionStateEvent` immutable class)

### B3. `WebKubiBleOptions.autoReconnect` 相当のコンストラクタ事前設定

  }

  abstract interface class KubiBle {

  - `lib/src/kubi_ble.dart` の `setAutoReconnect` 修正
  - `KubiBleImpl` (Phase 3) constructor に追加

- Decision: 案 A (`AutoReconnectConfig?` 一本化)

### B4. `setDefaultSpeed(MoveSpeed)` / `getDefaultSpeed()` / `MoveSpeed` (sealed)

Flutter 側 `MoveSpeedUniform`/`MoveSpeedPerAxis` 設計済みだが、`KubiBle` interface に `setDefaultSpeed`/`defaultSpeed` getter が**設計書 §3.2 にあるのに `lib/src/kubi_ble.dart` で未反映**。

- [x] **B4 決定**: 案 A 採用 — `MoveSpeed` 数値型を **`int`** に統一し interface に追加。
  ```dart
  sealed class MoveSpeed { const MoveSpeed(); }
  final class MoveSpeedUniform extends MoveSpeed {
    const MoveSpeedUniform(this.speed);
    final int speed;  // 1..100 (clamp 前は ArgumentError or 内部 clamp、A5.b で確定)
  }
  final class MoveSpeedPerAxis extends MoveSpeed {
    const MoveSpeedPerAxis({required this.pan, required this.tilt});
    final int pan;
    final int tilt;
  }

  abstract interface class KubiBle {
    void setDefaultSpeed(MoveSpeed speed);
    MoveSpeed get defaultSpeed;
    // 初期値: MoveSpeedUniform(KubiProtocol.defaultMoveSpeed = 100)
    // disconnect() 経由ではリセットされず、インスタンス寿命で保持 (TS と同じ)
  }
  ```
- 連動決定:
  - `MoveSpeedUniform.speed` / `MoveSpeedPerAxis.pan|tilt` を `double` → `int` に変更 (設計書 §4.1 破壊的更新)
  - B5 の `MoveSpecSynced.maxSpeed` も同じく `int` に統一 (B5 で連動)
  - A5.b の `clampSpeed` シグネチャは `int normalize(int value)` (null 受けは `setDefaultSpeed` etc. で別途 default fallback、`clampSpeed` 自体は null を受けない)
- 影響:
  - `docs/api-design.md` §3.2 / §4.1 を更新
  - `lib/src/kubi_ble.dart` に `setDefaultSpeed`/`defaultSpeed` 追加
- 理由: `KubiProtocol.minMoveSpeed/maxMoveSpeed` (int) と型一致。Flutter で型を狭く取る慣習。`MoveSpeedUniform(52.7)` をコンパイル時に弾ける。UI スライダー連携は呼び出し側で `.round()` を入れれば済む。
- Decision: 案 A (int 統一)

### B5. `moveTo` の `sync`/`maxSpeed` opt-in 切替（破壊的, v0.8）

TS: `MoveToExtendedOptions { speed?, signal?, settle?, sync?, maxSpeed? }`。
- `sync: true` → `maxSpeed` 必須、`speed` 排他 → arc-ratio + 補正テーブルで auto-sync
- `sync: false/省略` → `speed` を使用、`maxSpeed` 指定はエラー
- ランタイム例外 (`BleCommandError`) で誤組合せを拒否

Flutter 慣習的にはコンパイル時で弾けた方が安全。

- [x] **B5 決定**: 案 3 採用 — sealed `MoveSpec` + const factory で排他性をコンパイル時保証。
  ```dart
  sealed class MoveSpec {
    const MoveSpec();
    const factory MoveSpec.independent({MoveSpeed? speed}) = MoveSpecIndependent;
    const factory MoveSpec.synced({required int maxSpeed}) = MoveSpecSynced;
  }

  final class MoveSpecIndependent extends MoveSpec {
    const MoveSpecIndependent({this.speed});
    final MoveSpeed? speed;  // null → defaultSpeed (B4)
  }

  final class MoveSpecSynced extends MoveSpec {
    const MoveSpecSynced({required this.maxSpeed});
    final int maxSpeed;  // B4 int 統一に整合、1..100 (KubiProtocol.minMoveSpeed..maxMoveSpeed)
  }

  abstract interface class KubiBle {
    Future<MoveResult> moveTo({
      required PanTiltAngles target,
      MoveSpec spec = const MoveSpec.independent(),
      CancelToken? cancel,           // C7 で確定
      SettleOptions? settle,
    });
  }
  ```
- 影響:
  - `docs/api-design.md` §3.2 `moveTo` シグネチャ更新 (現状: `MoveSpeed? speed` → 新: `MoveSpec spec`)、§4.1 隣に `MoveSpec` 追加
  - `lib/src/kubi_ble.dart` の `moveTo` シグネチャ更新
  - `setTarget` は B5 対象外 (TS でも setTarget は sync を持たない、`speed` 直接受け取り)
- 理由:
  - TS が runtime で守ろうとしている排他性を Dart 3 sealed でコンパイル時保証
  - 設計書 §4.1 の `MoveSpeed` も sealed なので parallel 構造
  - `MoveSpec.synced(maxSpeed: 80)` が文法的に読みやすい
- Decision: 案 3 (sealed + const factory)

### B10. `subscribePosition` の中核改善（#10）

TS で 3 つの工夫:
1. **`setInterval` → `setTimeout` 再帰化**: 前回完了後に次スケジュール（idle skip 率 55% → 0%）
2. **`_lastObservedActual` cache**: `waitUntilSettled` poll 内で actual をキャッシュ、`subscribePosition` tick が lock 奪えなくても cache 配信（moveTo 中 skip 率 90% → <20%）
3. **batch atomic read**: `_pollPositionOnce` で 4 read を 1 lock で連続実行（tick 800ms → 400ms）

これは「subscribePosition が moveTo 中に死ぬ」問題の核心。Flutter 移植時に丸ごと採用しないと UX が劣化する。

- [x] **B10 決定**: 案 A 採用 — `Stream<PositionSnapshot> subscribePosition(SubscribePositionOptions options)` (broadcast stream)、内部実装は TS の 3 工夫を全部移植
  - **API 形状**:
    ```dart
    Stream<PositionSnapshot> subscribePosition(SubscribePositionOptions options);
    // 解除: stream.listen(...).cancel() で内部 timer 停止
    // listener 0 になったタイミングで poll timer を停止
    ```
  - **内部実装の 3 工夫（全部移植）**:
    1. **`Timer` 再帰化**: `Timer.periodic` 禁止、`Timer(duration, callback)` を毎回再スケジュール（前回完了後に次を組む）→ idle skip 率ゼロ化
    2. **`_lastObservedActual` cache**: `({DateTime ts, PanTiltAngles pos})?` を `KubiBleImpl` フィールドに保持。`moveTo` の settle poll が actual を読んだら更新。`_pollPositionOnce` で lock 取れない場合、cache が 1 秒以内なら配信、それ以上古ければ skip（TS と同じ閾値）
    3. **batch atomic read**: `_pollPositionOnce` 内で必要な register read（commanded/actual/both）を **1 lock 内で連続実行**。`source: both` 時は 4 read（pan-cmd, tilt-cmd, pan-act, tilt-act）を 1 lock で
  - **multiple listener の interval**: TS は entry ごとに interval を持ち全体最短で poll。Dart の Stream 単一化に伴い、`subscribePosition` を複数回呼ぶと **複数 Stream / 各 Stream に options** という形になる。内部 poll timer は **全 active subscription の最短 intervalMs** で動作（`min`）、最後の subscription cancel で停止
  - **`isMoving` の決定**: `_activeMove != null`（`Completer<MoveResult>` ベースの現在進行中 move 追跡フラグ）
  - **broadcast 仕様**: ListenError は subscription の `onError` ハンドラに渡る。詳細は B11 で
  - 影響: `lib/src/kubi_ble.dart` の `subscribePosition` シグネチャ確定、`docs/api-design.md` §3.3 更新

### B11. `subscribePosition` の `onError` オプション

### C6. 「Stream 購読時に現在値即 emit」を素の `Stream` で保証できない

設計書 §3.1 が謳う挙動を素の `Stream<T>` で保証する標準手段は無い。選択肢:
- (a) `rxdart` の `BehaviorSubject<T>` 依存追加
- (b) 独自 `ValueStream<T>` (内部 `StreamController.broadcast(onListen: ...)` + 最新値保持)
- (c) `ValueNotifier<T>` + `Stream<T>` の 2 系統公開（Flutter idiomatic）

- [x] **C6 決定**: 案 A 採用 — 個別 Stream は素の `Stream<T>` (broadcast、現在値即 emit なし)、現在値は `state` getter (C5) / `currentConnectionState` getter (B2) から取る責務分担

    - §3.1 の `connectionStateStream` Dartdoc から「購読時に現在値が即座に emit」を **削除**

  - **`state: ValueListenable<KubiState>` で吸収される現在値**: connectionState / commanded / actual / isMoving / lastError
  - **追加依存**: なし（`rxdart`/独自 `ValueStream` どちらも不要）

### C7. `CancelToken` 自前定義 vs Dart 慣習

TS は `AbortSignal`。Dart 標準にはないが `Completer<void>` ベースで作れる。
類似先行: `package:async` の `CancelableOperation`, `package:dio` の `CancelToken` 等。

- [x] **C7 決定**: 案 A 採用 — `lib/src/cancel_token.dart` に自前最小実装、TS `AbortSignal` と同じ使用感、依存追加なし
  ```dart
  /// `moveTo` / `waitUntilSettled` 等を外部から中断するためのトークン。
  ///
  /// TS の `AbortSignal` 相当。1 つの token は **1 回だけ** cancel 可能。
  /// 再利用したい場合は新しい token を生成すること。
  final class CancelToken {
    final _completer = Completer<void>();
    bool _cancelled = false;

    /// token を cancel する。冪等（複数回呼んでも安全）。
    void cancel() {
      if (_cancelled) return;
      _cancelled = true;
      _completer.complete();
    }

    /// cancel されたかどうかの同期チェック。
    bool get isCancelled => _cancelled;

    /// cancel されるまで完了しない Future。
    /// 内部の polling/await ループで `await Future.any([work, cancel.whenCancelled])` 等に使う。
    Future<void> get whenCancelled => _completer.future;
  }
  ```
  - **使用例**:
    ```dart
    final ct = CancelToken();
    final result = kubi.moveTo(target: ..., cancel: ct);
    // ... 後でキャンセル
    ct.cancel();
    // moveTo の Future は MoveResultCancelled で完了
    ```
  - **内部実装**: `moveTo` 内で `Future.any([_runMove(), cancel?.whenCancelled])` パターン、または polling ループで `if (cancel?.isCancelled == true) break` をチェック
  - **影響**: `lib/src/cancel_token.dart` 新設、`docs/api-design.md` §4.x に追加、`moveTo`/`waitUntilSettled`/`subscribePosition` 等のシグネチャに `CancelToken? cancel` を追加
  - **TS の `AbortSignal` 相当からの差分**: `AbortSignal` の `aborted: boolean` → `isCancelled`、`addEventListener('abort', cb)` → `whenCancelled` の Future。Dart の `await` パターンに合わせる

### C9. listener throw 隔離

TS は `try { cb(...) } catch { _emitDebug('listener:error', ...) }` で listener 例外を隔離。Flutter の `StreamController.broadcast` は default で例外が他 listener に伝播しないが、`onError` 経路を明示しないとデバッグできない。

- [x] **C9 決定**: 採用 — 全 stream で listener throw を `BleDebugEventType.listenerError` (B12) 経由で配信する実装パターンを設計書 §3.x に追記
  - **実装パターン**:
    ```dart
    void _safeNotify<T>(StreamController<T> ctrl, T event) {
      try {
        ctrl.add(event);
      } catch (e, st) {
        _emitDebug(BleDebugEvent(
          type: BleDebugEventType.listenerError,
          timestamp: DateTime.now(),
          message: 'listener threw on ${T.toString()}',
          detail: e,
        ));
      }
    }
    ```
  - ただし、Dart の `StreamController.broadcast` は default で listener の throw を **他 listener に伝播させない** 仕様（`onDone`/`onError` ハンドラに渡る）。明示の `try/catch` は必要に応じて。
  - **狙い**: TS と同じく「listener が throw しても他 listener や stream 自体は健全に動き続ける」+「ログとして観測可能」
  - **対象 stream**: `connectionStateStream`, `subscribePosition` の Stream, `moveEventStream`, `debugEventStream`
  - 影響: `docs/api-design.md` §3.x に listener throw 隔離パターンを追記、`KubiBleImpl` 実装時に該当処理を組み込む

### C10. `subscribePosition` を Stream で公開する場合の backpressure / skip セマンティクス

TS: drop-old、moveTo 中は cache 配信 or skip。
Stream で表現するには `StreamController(sync: false)` + 内部最新値保持を実装側で書く必要がある。

- [x] **C10 決定**: 採用 — `subscribePosition` の Stream の backpressure / skip セマンティクスを設計書に明文化
  - **明文化する内容**:
    1. **broadcast stream**: 複数 listener 可、各 listener は独立購読
    2. **drop-old セマンティクス**: 内部 buffer なし、tick で生成された snapshot は即座に全 listener に配信、配信後は破棄。listener が遅い場合の蓄積はしない
    3. **skip 条件**:
       - **gatt_busy + cache stale**: `_lastObservedActual` が 1 秒以上古い場合は **skip**（`BleDebugEventType.pollSkipped` に `detail: {reason: 'gatt_busy'}`）、stream に配信なし、次の tick へ
       - **gatt_busy + cache fresh** (< 1 秒): cache から actual を配信（`isMoving: true`）
       - **lock available**: batch read 実行、commanded/actual を取得して配信
    4. **エラー時挙動** (B11 と整合): register read timeout 等は `controller.addError(...)` で配信、stream は close しない、次の tick をスケジュール
    5. **subscription cancel**: 全 subscription が cancel されたら内部 Timer を停止、新規 listen で再開
  - 影響: `docs/api-design.md` §3.3 に「subscribePosition のセマンティクス」サブセクション新設、B10 の実装方針と一体化して記述

### C13. `MoveSpeed` ファクトリ

TS は `100` または `{ pan: 100, tilt: 30 }` という JS 自然構文。Dart で `MoveSpeedUniform(100)` / `MoveSpeedPerAxis(pan: 100, tilt: 30)` は冗長。
- 案 1: `MoveSpeed.uniform(100)` / `MoveSpeed.perAxis(pan: 100, tilt: 30)` factory に統一
- 案 2: 現状維持

- [x] **C13 決定**: 案 A 採用 — `MoveSpeed` を sealed (B4) のまま、const factory に統一
  ```dart
  sealed class MoveSpeed {
    const MoveSpeed();
    const factory MoveSpeed.uniform(int speed) = MoveSpeedUniform;
    const factory MoveSpeed.perAxis({required int pan, required int tilt}) = MoveSpeedPerAxis;
  }

  final class MoveSpeedUniform extends MoveSpeed {
    const MoveSpeedUniform(this.speed);
    final int speed;  // B4 で int 確定
  }

  final class MoveSpeedPerAxis extends MoveSpeed {
    const MoveSpeedPerAxis({required this.pan, required this.tilt});
    final int pan;
    final int tilt;
  }
  ```
  - **使用例**:
    ```dart
    kubi.setDefaultSpeed(MoveSpeed.uniform(80));
    kubi.moveTo(target: ..., spec: MoveSpec.independent(speed: MoveSpeed.perAxis(pan: 100, tilt: 30)));
    ```
  - **TS との整合**: TS の `100` は `MoveSpeed.uniform(100)`、`{ pan: 100, tilt: 30 }` は `MoveSpeed.perAxis(pan: 100, tilt: 30)`
  - **subclass 直接構築も残る**: `MoveSpeedUniform(100)` も `final class` のため動作するが、Dartdoc では factory 経由を推奨表記
  - **switch narrow**: sealed なので `switch (speed) { case MoveSpeedUniform(:final speed): ...; case MoveSpeedPerAxis(:final pan, :final tilt): ...; }` で exhaustive
  - **値域 validation**: 各 subclass のコンストラクタで `assert(speed >= minMoveSpeed && speed <= maxMoveSpeed)` を組む（A5.b で残した issue「`clampSpeed` 相当が欠落」の最小対応 = ArgumentError）
  - 影響: `lib/src/types/move_speed.dart` (新設) / `docs/api-design.md` §4.1 を更新

## D. universal_ble 固有の懸念（実装前に検証必須）

> **D1-D4 一括方針** (B17 と統合): Phase 3 着手前に **universal_ble 公式ドキュメント読み込み + 各プラットフォーム挙動の論理整理 + ドラフト設計作成** を 1 つのタスクとして実施。`queue/universal-ble-investigation.md` を新設し、結果を表形式で記録。実機検証は Phase 3 中に並行実施。

### D1. `requestDevice` の Web vs Native 責務差

Web: browser picker 1 アクションで完結。
Native (Android/iOS/Windows/macOS/Linux): 「scan 開始 → 結果リスト UI → ユーザー選択 → scan 停止」の責務をどこに置くか。

`KubiBle.requestDevice()` がプラットフォームを跨いで `Future<KubiDevice>` 1 個を返すなら、Native 実装は内部で UI を開く必要がある。これは package 責務として重い。

- [x] **D1 決定**: 一括調査タスクへ（下記 universal-ble-investigation 参照）。**暫定方針** (調査前): 案 B 採用 — `Stream<KubiDevice> scanForDevices()` を全 platform で公開、Web は内部で `requestDevice` を呼んで 1 件返す stream として実装、Native は scan 結果を流す。アプリ側で UI を作る。Flutter プラグイン慣習に整合。最終決定は universal_ble 実装パス確認後。

### D2. universal_ble の notify 順序保証

TS の `_readRegister` プロトコルは、`writeValueWithResponse(motorId, addr)` 直後に `characteristicvaluechanged` で 1:1 で応答が返ることに強く依存。universal_ble の `onValueChange` が同等の保証を持つか実機検証必須。

- [x] **D2 決定**: Phase 3 着手前にドキュメントで挙動を確認、Phase 3 早期に実機検証で確定。**実装方針** (調査前): `_readRegister` 内で「1 回の write → 直後 1 回の notify」契約を `Completer<Uint8List>` ベースのキューで管理し、順序保証されない場合に備えて **request id (motorId+addr) を notify payload と照合** するガードを入れる（TS は payload に register address が乗ることを利用して照合）。

### D3. `writeValueWithResponse` 相当

universal_ble の `writeValue` で `BleOutputProperty.withResponse` を指定する。TS は「`writeValueWithResponse` 非対応ブラウザは fail-fast」(`BleUnavailableError`) しているが、universal_ble はプラットフォーム毎に内部で吸収しているはず。Web プラットフォームでの fail-fast 条件を Flutter 側でも保持するか検討。

- [x] **D3 決定**: 一括調査タスクへ。**実装方針** (調査前): `BleOutputProperty.withoutResponse` (PAN/TILT/MOTOR_DATA) と `withResponse` (regRead*) を **明示指定**。Web で `withResponse` 非対応の場合は universal_ble が例外を投げる想定で、`KubiBle` 側で `BleUnavailableError` に変換するハンドラを `connect` 直後の能力チェックで実装。

### D4. `tryAutoConnect` の permission モデル差

Web: `getDevices()` で過去に permission 付与されたデバイスのみ。
Android: companion device manager / bluetooth-connect runtime permission。
iOS: `retrievePeripheralsWithIdentifiers` 相当。
universal_ble はこれを `getSystemDevices` で抽象化していると想定だが、`name` フィルタが効くか要確認。

- [x] **D4 決定**: 一括調査タスクへ。**実装方針** (調査前): `getSystemDevices(withServices: [servoServiceUuid])` を呼び、戻り値リストを `name?.startsWith('kubi')` で filter（namePrefix 大文字小文字区別あり、TS 仕様 §1.3 と整合）。filter 後最初の 1 件を返す（複数台環境は別 issue）。permission 不在の場合は `BleNotConnectedError` ではなく `null` を返す（TS の `tryAutoConnect: KubiDevice | null` と整合）。

## G. Phase 2.5 Decision Traceability (D1-D5)

> **SSOT**: `queue/phase-2.5-universal-ble-investigation.md` (調査根拠 + 推奨案の論拠)。本節は決定事項のサマリと反映先のみ。

### D1. `requestDevice` 実装 — 案 C 採用

`Stream<KubiDevice> scan({Duration? timeout, ScanFilter? filter})` を新設し、`requestDevice({Duration timeout = 5s})` は `scan().first.timeout(...)` の convenience に再定義。

**反映先:**
- 設計書 §3.2 (`scan` / `requestDevice` 行追加)
- `lib/src/kubi_ble.dart` (`scan` / `requestDevice` シグネチャ更新)
- CHANGELOG Added/Changed
- 暫定: `KubiScanFilter` の独自定義 vs universal_ble `ScanFilter` 再 export は **Phase 3 で再判断** (当面 `withNamePrefix: ['kubi']` 固定実装)

### D2. GATT queue 二重化 — 案 Y 採用

`UniversalBle.queueType = QueueType.perDevice` を `KubiBleImpl` constructor で設定し、write/read 直列化は universal_ble の `BleCommandQueue` に委譲。我々の self-lock は「moveTo cancel-on-newer」「latest-value buffer」「subscribe poll skip」のアプリ層ロジックのみ。

**反映先:** 設計書 §5.1 (書き換え済)、CHANGELOG。Phase 3 実装で `KubiBleImpl` constructor に反映予定。

### D3. `connectionStream` 中間状態の補完

universal_ble の `connectionStream` は bool のみ (中間状態を流さない) なので、`KubiBleImpl` 内で手動 emit。`UniversalBle.onConnectionChange = (id, isConnected, error)` callback で `error` を取得し、`error != null → DisconnectReason.error` / `error == null && !isConnected → DisconnectReason.deviceLost` にマップ。

**反映先:** 設計書 API 変更なし (実装ロジックで吸収)、Phase 3 KubiBleImpl で実装。

### D4. Availability 監視

`UniversalBle.availabilityStream` を `KubiBleImpl` で listen し、`poweredOff / unauthorized / resetting` 検知時に内部 `disconnect()` + `ConnectionStateEvent(disconnected, deviceLost)` emit。`unsupported / unauthorized` 中は auto-reconnect を abandon。

**反映先:**
- 設計書 §5.8 「Availability 監視」新節
- 設計書 §3.2 `availabilityStream` 行
- 設計書 §3.6 `BleAvailability` enum 追加
- `lib/src/types/ble_availability.dart` 新設、`lib/src/kubi_ble.dart` getter 追加
- CHANGELOG Added

### D5. Web `tryAutoConnect` 既知制約

universal_ble v1.2.0 は `navigator.bluetooth.getDevices()` を wrap していないため、Web では `tryAutoConnect()` は常に `null` を返す。**既知制約として明文化** (将来 universal_ble 側が対応すれば追従)。

**反映先:**
- 設計書 §4.3 末尾の D5 注記
- `lib/src/kubi_ble.dart` `tryAutoConnect` の dartdoc
- CHANGELOG Changed
